- `chapter_2/s02_tool_use.py`

**脚本层代码结构**：

```python
if __name__ == "__main__":
    history = [{"role": "system", "content": SYSTEM_PROMPT}]
    
    while True:
        query = input("User: ")
        if query.lower() in ("q", "exit"):
            break
        
        history.append({"role": "user", "content": query})
        state = LoopState(messages=history)
        agent_loop(state)
        history = state.messages
        
        last_message = state.messages[-1]
        raw_content = (
            last_message.content
            if hasattr(last_message, "content")
            else last_message.get("content", "")
        )
        final_text = extract_text(raw_content)
        if final_text:
            print(f"[最终回复] {final_text}")
        print()
```

**特点**：
- 简单的 `while input()` 循环
- 基础的历史记录管理
- 统一的输出处理逻辑（兼容 `ChatCompletionMessage` 对象和字典）

**演变说明**：
- 这四章的脚本层逻辑基本一致
- 主要变化发生在核心逻辑层（如 chapter_4 引入 TodoManager），但脚本层入口代码保持不变

### Capiter 14：事件驱动架构

**文件**：`chapter_14/s14_cron_scheduler.py`

**架构重大变化**：不再使用简单的 `while input()` 循环，引入多线程事件队列机制。

**核心改动**：

```python
# ── 启动事件驱动线程 ──────────────────────────────────
_event_queue: Queue = Queue()
_stop_cron_watcher = threading.Event()
_input_ready = threading.Event()
_input_ready.set()   # 初始状态：允许立刻显示提示符

threading.Thread(
    target=input_reader, args=(_event_queue, _input_ready), daemon=True, name="input_reader"
).start()
threading.Thread(
    target=cron_watcher, args=(_event_queue, _stop_cron_watcher),
    daemon=True, name="cron_watcher"
).start()

# ── 主循环：阻塞在 event_queue 上，由任意事件唤醒 ───────────────
try:
    while True:
        event_type, content = _event_queue.get()  # 任意事件到来即唤醒

        # ── quit ────────────────────────────────────────────────────
        if event_type == "quit":
            _stop_cron_watcher.set()
            scheduler.stop()
            break

        # ── cron 触发：无需用户输入，直接注入历史并运行 agent_loop ──
        if event_type == "cron":
            print()
            history.append({
                "role": "user",
                "content": f"<cron-notification>\n{content}\n</cron-notification>",
            })
            state = LoopState(messages=history)
            agent_loop(state, compact_state)
            history = state.messages
            # ... 输出处理 ...
            _input_ready.set()   # 恢复提示符
            continue

        # ── user 输入 ────────────────────────────────────────────────
        query = content
        # ... 处理用户输入和命令 ...
```

**新增命令**：
- `/cron`：查看定时任务列表
- `/test`：触发测试事件，验证 cron 通知机制

**资源清理逻辑**：

```python
finally:
    _input_ready.set()       # 解除 input_reader 的 wait() 阻塞
    _stop_cron_watcher.set()
    scheduler.stop()
    # 还原 stdout/stderr
    sys.stdout = sys.__stdout__
    sys.stderr = sys.__stderr__
    if _log_file is not None:
        try:
            _log_file.close()
        except Exception:
            pass
    # 还原终端状态
    if _saved_tty is not None:
        try:
            termios.tcsetattr(sys.stdin, termios.TCSADRAIN, _saved_tty)
        except Exception:
            pass
    print("\n[Bye]")
```

**变化说明**：
- **架构变更**：从同步 `input()` 循环变为异步事件驱动模型
- **多线程**：`input_reader` 线程负责读取用户输入，`cron_watcher` 线程负责定时任务触发
- **事件队列**：主循环阻塞在 `_event_queue.get()` 上，处理三种事件类型：
  - `quit`：退出会话
  - `cron`：定时任务触发（无需用户输入）
  - `user`：用户输入
- **会话日志**：使用 `_Tee` 类将 stdout/stderr 重定向到日志文件
- **终端状态保护**：保存和还原 termios 设置，防止 Ctrl+C 退出后终端状态异常

## 演变总结

### 架构演进路线

1. **阶段 1（chapter_1-4）**：基础交互循环
   - 简单的 `while input()` 同步循环
   - 无运行时命令，仅支持 `q`/`exit` 退出

2. **阶段 2（chapter_5-9）**：功能增强期
   - 彩色输出（chapter_5）
   - 上下文压缩状态保持（chapter_6）
   - 权限模式控制（chapter_7）
   - 动态授权（chapter_8）
   - 记忆系统（chapter_9）
   - 斜杠命令系统逐步建立

3. **阶段 3（chapter_10-13）**：系统完善期
   - 动态提示词构建（chapter_10）
   - 动态提示符 UI（chapter_11）
   - 任务系统适配（chapter_12）
   - 斜杠命令：`/mode`、`/rules`、`/allow`、`/memories`、`/clear`

4. **阶段 4（chapter_14）**：架构升级
   - 从同步循环升级为事件驱动架构
   - 引入多线程和事件队列
   - 支持定时任务自动触发（无需用户输入）
   - 新增 `/cron`、`/test` 命令

5. **阶段 5（chapter_18_2-19_2）**：生态扩展
   - Worktree 任务隔离（chapter_18_2）
   - MCP 插件系统（chapter_19_2）

### 关键技术点

1. **事件驱动模型**（chapter_14）：
   - 主循环阻塞在事件队列上，不再主动轮询用户输入
   - 支持多种事件源：用户输入、定时任务、未来可扩展其他事件
   - 通过 `_input_ready` 事件控制提示符显示时机，避免提示符打断 agent 输出

2. **终端状态保护**（chapter_14）：
   - 启动时保存 termios 设置
   - 退出时在 `finally` 块中还原，防止 Ctrl+C 导致终端异常
   - 使用 `_Tee` 类实现日志文件重定向

3. **MCP 插件架构**（chapter_19_2）：
   - 插件扫描 → 服务器连接 → 工具注册的初始化流程
   - 独立的 MCP 权限门控（`mcp_gate`）
   - 统一的工具池构建（`build_mcp_tool_pool`）

## 结语

从 chapter_1 到 chapter_19_2，脚本层经历了从简单交互循环到复杂事件驱动架构的演变。核心改动包括：

1. 交互模式：同步 `input()` → 异步事件队列
2. 命令系统：无命令 → 10+ 斜杠命令
3. 状态管理：简单历史记录 → 压缩状态、任务状态、记忆状态、worktree 状态
4. 扩展能力：单体脚本 → MCP 插件系统

脚本层作为用户与核心逻辑的桥梁，逐步承担起状态初始化、命令解析、事件分发、资源清理等职责，为上层功能提供稳定的运行环境。
